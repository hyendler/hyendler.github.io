Dynamically Static (or the other way around?)

Dynamic vs. Static typed languages

12/22/15

As per usual, given any concept to research in the computer science world leads down the tangled Alice-In-Wonderland type rabbit hole. First thing: when researching dynamic vs. static typed languages, pops up the concepts of compilers, interpreters, variable interpretation at run time vs. compile time, and scripting languages. Huh?

So, to start with a few concepts and vocabulary:

  • Compiler: a computer program that takes code and alchemicalizes it into a different language. Now, why would this ever happen? Seems like a lot more work. According to Wikipedia, normally compilers "translate source code from a high-level programming language to a lower level language (e.g., assembly language or machine code)". Compilers have a long history within the development of computer science.
  • Compiled language: a coding language that requires a compiler. So to have it run, it needs to be compiled first. Examples: Java, C, C++
  • Machine code: the language that a compiled code gets transformed to.

Java looks something like this:

public class HelloWorld {
  public static void main(String[] args) {
    System.out.println("Hello World !");
  }
}

And when compiled, looks something like this (except 200 lines of it)

  • Interpreter: a computer program that runs the code right then and there, with no compilation. (However, some light code translation might happen right as the code is being executed.
  • Interpreted language: a language that can be executed right away. Examples: Ruby, Javascript, Python
  • Scripting languages: a scripting language is a type of interpreted language (from what I could gather). For extra interesting morsels, scripts are a list of executable series of instructions, and scripting languages serve the purpose of mediating between programs.

So now on to the actual definitions of the topic of this week, dynamic and static:

  • Static-typed language: When a language is compiled, the variable type is explicity declared.
  • Variable type: In static-typed languages such as Java, there are two different variable types: primitive and object/reference. Primitive types, for example, are categorized as integer or float or string, etc. But here is the catch: the program does not know which type of variable you are giving it, so you must explicitly declare it, like this:
    • int number = 100;
String word = "hell yeah";
  • Dynamic-typed language: The variable is interpreted at execution, not during compiliation. As a result, the languages do not require variable type declaration, such as in Ruby:
    • number = 100
word = "hell yeah"

But why is this a big enough deal in the first place? Well, static-typed languages are more performance efficient, but dynamic-typed languages are easier to write from a coder's perspective, because you don't have to track the variable type. However, that does mean that things can execute in a dynamic language that wouldn't normally execute under a static language, leading to errors during runtime. As far as a more in-depth discussion of the actual pros and cons of each, as a Stack Overflower stated so on point: "This question is too subjective".
